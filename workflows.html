CI
Tests & Qualité
ci.yml · 3 jobs · ~4-6 min
push dev/main PR vers dev/main/release/hotfix
1
PHPUnit Tests
job: tests
toujours actif parallèle avec lint

Lance les 24 tests PHPUnit sur une vraie base MySQL 8.0 (pas de mock). C'est le job le plus critique : s'il échoue, SonarCloud et le CD ne se déclenchent pas. Il génère aussi le rapport de couverture de code (coverage.xml) transmis à SonarCloud.

  1. 1Service MySQL 8.0 — GitHub Actions démarre un container MySQL parallèlement au job. Healthcheck toutes les 10s avant de continuer. Base : resources_test / user : app
  2. 2Setup PHP 8.4 + PCOV — shivammathur/setup-php@v2 installe PHP 8.4 avec les extensions pdo, pdo_mysql, intl, mbstring, opcache et le driver de couverture PCOV (plus rapide que Xdebug)
  3. 3Cache Composer — le dossier vendor/ est mis en cache avec la clé composer-{hash(composer.lock)}. Si composer.lock n'a pas changé, les dépendances ne sont pas retéléchargées
  4. 4composer install — avec --optimize-autoloader et --no-scripts pour éviter les effets de bord
  5. 5Génération des clés JWT RSA-4096 — openssl genpkey + openssl rsa -pubout. La passphrase vient de secrets.JWT_PASSPHRASE. Sans ces clés, Symfony ne peut pas signer les tokens → tous les tests d'authentification échoueraient
  6. 6Création du .env minimal — Symfony cherche un fichier .env au démarrage (bootEnv()). On crée echo "APP_ENV=test" > .env — les vraies valeurs viennent des variables d'env du runner (GitHub Secrets)
  7. 7Cache clear + migrations — bin/console cache:clear puis doctrine:migrations:migrate — crée les tables dans resources_test
  8. 8Fixtures — doctrine:fixtures:load --append. Le flag --append évite le TRUNCATE (qui peut bloquer sur les FK). Injecte les données de test : utilisateurs, ressources, catégories
  9. 9PHPUnit — --coverage-clover coverage.xml --stop-on-failure. S'arrête au premier test échoué. Génère le rapport XML de couverture
  10. 10Upload coverage artifact — coverage.xml uploadé comme artifact GitHub Actions (TTL 1 jour) pour être téléchargé par le job sonarcloud
Extrait YAML ci.yml · job: tests
services:
  mysql:
    image: mysql:8.0
    env:
      MYSQL_DATABASE: resources_test
      MYSQL_USER: app
      MYSQL_PASSWORD: app
    options: >-
      --health-cmd="mysqladmin ping -u app -papp"
      --health-interval=10s
      --health-retries=5

steps:
  - name: Setup PHP 8.4
    uses: shivammathur/setup-php@v2
    with:
      php-version: "8.4"
      coverage: pcov

  - name: Generate JWT keys
    run: |
      openssl genpkey -algorithm RSA \
        -out config/jwt/private.pem \
        -aes256 -pass pass:${{ secrets.JWT_PASSPHRASE }}
      openssl rsa -pubout \
        -in config/jwt/private.pem \
        -out config/jwt/public.pem \
        -passin pass:${{ secrets.JWT_PASSPHRASE }}

  - name: Run tests
    run: php bin/phpunit
      --coverage-clover coverage.xml
      --stop-on-failure
2
Lint & Static Analysis
job: lint
toujours actif parallèle avec tests

Vérifie la syntaxe PHP, les fichiers de configuration YAML et le container Symfony sans lancer les tests. S'exécute en parallèle avec le job tests pour réduire le temps total du pipeline. N'a pas besoin de base de données.

  1. 1PHP Syntax check — find src/ -name "*.php" -exec php -l {} — vérifie que chaque fichier PHP est parseable. Attrape les erreurs de syntaxe avant même que les tests ne tournent
  2. 2Lint YAML — php bin/console lint:yaml config/ --parse-tags — valide tous les fichiers YAML de config Symfony (services.yaml, security.yaml, routes.yaml…)
  3. 3Lint container — php bin/console lint:container — vérifie que tous les services définis dans le DIC (Dependency Injection Container) Symfony sont bien configurés et que leurs dépendances existent
Note : le lint utilise des valeurs factices pour les secrets (ci_dummy_secret) car il ne fait pas tourner l'application, juste l'analyser. Pas de connexion BDD nécessaire.
Extrait YAML ci.yml · job: lint
lint:
  name: Lint & Static Analysis
  runs-on: ubuntu-latest
  # pas de needs: → parallèle avec tests

  env:
    APP_ENV: dev
    APP_SECRET: ci_dummy_secret_for_lint_only
    # Valeurs factices — lint ne fait pas tourner l'app

  steps:
    - name: PHP Syntax check
      run: |
        find src/ -name "*.php" \
          -exec php -l {} \; \
          | grep -v "No syntax errors" || true

    - name: Lint YAML
      run: php bin/console lint:yaml config/ --parse-tags

    - name: Lint container
      run: php bin/console lint:container --no-debug
3
SonarCloud Analysis
job: sonarcloud
needs: tests skip si Dependabot

Analyse la qualité et la sécurité du code source via la plateforme SonarCloud. Ne peut démarrer qu'après la fin du job tests car il a besoin du rapport de couverture coverage.xml produit par PHPUnit.

  1. 1Checkout avec fetch-depth: 0 — récupère l'historique git complet. SonarCloud analyse les nouveaux commits par rapport à la branche de base — sans l'historique complet, il ne peut pas calculer le différentiel
  2. 2Download artifact coverage — télécharge coverage.xml uploadé par le job tests. SonarCloud lit ce fichier pour connaître le taux de couverture de code ligne par ligne
  3. 3SonarCloud Scan — SonarSource/sonarcloud-github-action analyse le code, détecte les bugs, code smells, vulnérabilités, et calcule le Quality Gate. Quality Gate configuré : 0 nouvelles vulnérabilités, 0 bugs bloquants, couverture > 70%
Conditionnel : if: github.actor != 'dependabot[bot]' — Dependabot n'a pas les permissions pour accéder à SONAR_TOKEN (secret de dépôt). Le scan est ignoré sur ses PRs.
Extrait YAML ci.yml · job: sonarcloud
sonarcloud:
  name: SonarCloud Analysis
  needs: tests
  if: github.actor != 'dependabot[bot]'

  steps:
    - name: Checkout
      uses: actions/checkout@v4
      with:
        fetch-depth: 0
        # historique complet requis pour le diff

    - name: Download coverage report
      uses: actions/download-artifact@v4
      with:
        name: coverage
        # coverage.xml uploadé par le job tests

    - name: SonarCloud Scan
      uses: SonarSource/sonarcloud-github-action@master
      env:
        GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        SONAR_TOKEN:  ${{ secrets.SONAR_TOKEN }}

1
Composer Audit — CVE PHP
job: composer-audit
toujours actif OWASP A06

Vérifie les dépendances Composer contre la base de données des advisories de sécurité PHP (Packagist Security Advisories). Détecte les packages avec des CVE connues avant de builder ou déployer.

  1. 1composer audit --format=json — produit un rapport JSON listant les packages vulnérables avec leur CVE, sévérité (CRITICAL/HIGH/MEDIUM/LOW), lien advisory
  2. 2Script Python de filtre — parse le JSON, compte les advisories de sévérité CRITICAL ou HIGH. Si le count est > 0, affiche chaque vulnérabilité et quitte avec code 1 → le job échoue et bloque le CD
  3. 3Upload artifact — le rapport JSON est conservé même si le job échoue (if: always()) pour pouvoir être consulté et archivé
Extrait YAML security.yml · composer-audit
    - name: Composer audit
      run: composer audit --format=json 2>&1 | tee composer-audit.json || true

    - name: Check for critical vulnerabilities
      run: |
        CRITICAL=$(cat composer-audit.json | python3 -c "
        import json,sys
        data=json.load(sys.stdin)
        advisories=data.get('advisories',{})
        critical=[a for pkg in advisories.values()
                  for a in pkg
                  if a.get('severity','').lower()
                  in ['critical','high']]
        print(len(critical))
        " || echo "0")
        echo "Critical/High found: $CRITICAL"
        if [ "$CRITICAL" -gt "0" ]; then
          echo "❌ Vulnérabilités CRITICAL/HIGH détectées!"
          exit 1
        fi
2
Trivy — Scan image Docker
job: trivy-image
main/dev push seulement OWASP A06

Build l'image Dockerfile.prod (multi-stage Alpine) et la scanne avec Trivy pour détecter les CVE dans l'OS Alpine, les bibliothèques système et les paquets PHP. Les résultats sont remontés dans l'onglet Security du repo GitHub.

  1. 1docker build -f Dockerfile.prod — build l'image de production complète (les 2 stages). Taggée ressources-api:scan localement
  2. 2Trivy scan (table) — format lisible dans les logs. exit-code: 1 → le job échoue si une CVE CRITICAL ou HIGH non patchée est trouvée. ignore-unfixed: true ignore les CVE sans correctif disponible
  3. 3Trivy scan (SARIF) — format standardisé pour GitHub. Couvre CRITICAL/HIGH/MEDIUM. S'exécute if: always() même si le scan table a échoué, pour avoir le rapport complet
  4. 4Upload SARIF — github/codeql-action/upload-sarif pousse les résultats dans l'onglet Security → Code scanning du repo GitHub
Conditionnel : if: github.ref == 'refs/heads/main' || 'refs/heads/dev' — ne tourne pas sur les PRs car builder une image complète prend 3-5 min. Les PRs sont couvertes par composer-audit et trivy-code.
Extrait YAML security.yml · trivy-image
    - name: Build image for scanning
      run: docker build -f Dockerfile.prod -t ressources-api:scan .

    - name: Trivy scan — CRITICAL & HIGH
      uses: aquasecurity/trivy-action@master
      with:
        image-ref:     ressources-api:scan
        format:        table
        exit-code:     "1"
        severity:      CRITICAL,HIGH
        ignore-unfixed: true

    - name: Trivy scan — SARIF (GitHub Security tab)
      uses: aquasecurity/trivy-action@master
      if: always()
      with:
        format:  sarif
        output:  trivy-results.sarif
        severity: CRITICAL,HIGH,MEDIUM

    - name: Upload SARIF to GitHub Security
      uses: github/codeql-action/upload-sarif@v4
      if: always()
3
Trivy — Scan code source
job: trivy-code
toujours actif parallèle

Scanne le code source (filesystem) plutôt que l'image Docker. Détecte trois catégories de problèmes : secrets hardcodés, misconfigurations (fichiers de config dangereux), et vulnérabilités dans les fichiers de dépendances.

  1. 1scan-type: fs — mode filesystem, scanne le repo directement sans builder d'image
  2. 2scanners: secret,misconfig,vuln — 3 types simultanément : clés API dans le code, fichiers .env commités, vulnérabilités déclarées dans composer.json
  3. 3Upload SARIF — résultats remontés dans GitHub Security tab
Extrait YAML security.yml · trivy-code
    - name: Trivy filesystem scan
      uses: aquasecurity/trivy-action@master
      with:
        scan-type: fs
        scan-ref:  .
        format:    sarif
        output:    trivy-fs-results.sarif
        scanners:  secret,misconfig,vuln
        # secret  → clés API/tokens dans le code
        # misconfig → .env commités, permissions
        # vuln    → dépendances composer.json
4
Gitleaks — Détection de secrets
job: gitleaks
toujours actif parallèle

Scanne l'intégralité de l'historique git (tous les commits, pas seulement le dernier) à la recherche de secrets commités par erreur : tokens API, mots de passe, clés privées. Complémentaire à Trivy-code qui ne scanne que l'état actuel.

  1. 1fetch-depth: 0 — checkout de l'historique complet. Sans ça, seul le dernier commit est analysé — un secret commité il y a 3 mois puis supprimé dans un commit suivant passerait inaperçu
  2. 2gitleaks-action@v3 — détecte plus de 150 patterns (AWS keys, GitHub tokens, Supabase keys, JWT secrets, mots de passe dans les URLs de connexion…)
Extrait YAML security.yml · gitleaks
gitleaks:
  name: Gitleaks — Secret detection
  runs-on: ubuntu-latest

  steps:
    - name: Checkout
      uses: actions/checkout@v4
      with:
        fetch-depth: 0
        # TOUT l'historique git — pas seulement HEAD

    - name: Gitleaks scan
      uses: gitleaks/gitleaks-action@v3
      env:
        GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
5
OWASP Dependency Check
job: owasp-dependency-check
main push + nuit seulement NIST NVD

Analyse plus approfondie que composer-audit : consulte la base NIST NVD (National Vulnerability Database) complète pour croiser les dépendances avec toutes les CVE connues, y compris les bibliothèques transitives non déclarées directement dans composer.json.

  1. 1Dependency-Check_Action — télécharge la NVD (5-15 min selon le cache), analyse récursivement toutes les dépendances directes et transitives
  2. 2--failOnCVSS 7 — échoue si une CVE avec un score CVSS ≥ 7.0 (HIGH/CRITICAL) est trouvée. Le score CVSS va de 0 à 10
  3. 3--enableRetired — inclut aussi les packages retirés (dépréciés), pas seulement ceux avec CVE active
  4. 4format: ALL — génère les rapports en HTML, JSON, XML, CSV → uploadés comme artifact
Pourquoi pas sur chaque PR ? Le téléchargement de la NVD prend 5-15 minutes et génère des milliers de faux positifs non applicables à PHP. composer-audit (5s) + Trivy couvrent les cas critiques. OWASP DC sert de filet de sécurité nocturne plus exhaustif.
Extrait YAML security.yml · owasp-dependency-check
owasp-dependency-check:
  if: github.event_name == 'schedule'
      || github.ref == 'refs/heads/main'
  # NVD = 5-15 min de téléchargement
  # On évite ça sur chaque PR

  steps:
    - name: OWASP Dependency Check
      uses: dependency-check/Dependency-Check_Action@main
      with:
        project: ressources_api
        path:    .
        format:  ALL
        args: >
          --enableRetired
          --failOnCVSS 7
CD
Build & Deploy
cd.yml · 3 jobs · ~5-10 min
après CI + Security ✓ sur main workflow_dispatch (prod)
1
Build & Push Docker
job: build
si CI + Security passés GHCR

Construit l'image Docker de production (Dockerfile.prod, multi-stage Alpine) et la pousse sur GitHub Container Registry (GHCR) avec deux tags : le SHA court du commit et :latest. L'image est immuable et traçable.

  1. 1Condition workflow_run — le CD ne démarre que si les workflows CI et Security ont tous les deux terminé avec conclusion == 'success' sur main. Si l'un échoue, le CD ne se lance pas
  2. 2Login GHCR — docker/login-action avec secrets.GHCR_USERNAME + secrets.GHCR_TOKEN. Le token est un Personal Access Token GitHub avec scope write:packages
  3. 3Docker Buildx — active le builder multi-platform de Docker (supporte arm64/amd64). Nécessaire pour la compatibilité cross-platform
  4. 4docker/metadata-action — génère automatiquement les tags : sha-abc1234f (SHA court) + :latest sur la branche par défaut. Expose steps.meta.outputs.version pour les jobs de déploiement
  5. 5Build & push — build depuis Dockerfile.prod, push vers ghcr.io/txmy-elu/ressources_api. Cache GHA utilisé pour réutiliser les layers entre les builds
  6. 6Output image_tag — le SHA est passé en outputs.image_tag pour que les jobs de déploiement sachent quelle version déployer
Extrait YAML cd.yml · job: build
on:
  workflow_run:
    workflows:
      - "CI — Tests & Quality"
      - "Security — CVE & OWASP"
    types: [completed]
    branches: [main]

jobs:
  build:
    if: >
      github.event_name == 'workflow_dispatch' ||
      github.event.workflow_run.conclusion == 'success'

    outputs:
      image_tag: ${{ steps.meta.outputs.version }}

    steps:
      - name: Docker metadata
        id: meta
        uses: docker/metadata-action@v6
        with:
          images: ghcr.io/${{ github.repository }}
          tags: |
            type=sha,prefix=,format=short
            type=raw,value=latest,enable={{is_default_branch}}

      - name: Build & push
        uses: docker/build-push-action@v5
        with:
          file:       Dockerfile.prod
          push:       true
          cache-from: type=gha
          cache-to:   type=gha,mode=max
2
Deploy — Préprod
job: deploy-preprod
needs: build PREPROD_ENABLED = true

Se connecte au serveur de préprod en SSH et déploie la nouvelle image Docker sans toucher à la base de données ni aux autres services. Zéro downtime : seul le container PHP est redémarré (--no-deps).

  1. 1environment: preprod — GitHub Environment avec protection rules optionnelles (reviewers requis, wait timer). Permet de tracer qui a approuvé quel déploiement
  2. 2SSH via appleboy/ssh-action — utilise secrets.PREPROD_SSH_KEY (clé RSA privée du runner). Le runner se connecte en tant que secrets.PREPROD_USER sur secrets.PREPROD_HOST
  3. 3docker login GHCR — le VPS doit s'authentifier pour puller l'image privée depuis GHCR
  4. 4export IMAGE_TAG — récupère le SHA du commit depuis needs.build.outputs.image_tag pour puller exactement cette version
  5. 5docker compose pull php — télécharge uniquement la nouvelle image PHP (pas mysql, nginx, monitoring)
  6. 6docker compose up -d --no-deps php — redémarre uniquement le container PHP avec la nouvelle image. La BDD, Nginx et le monitoring restent en place → zéro downtime
  7. 7doctrine:migrations:migrate — applique les nouvelles migrations SQL si présentes. --no-interaction pour ne pas attendre de confirmation
  8. 8docker image prune -f — supprime les anciennes images pour libérer l'espace disque du VPS
Actuellement désactivé : PREPROD_ENABLED est à false (Settings › Variables). Pas de VPS dans ce projet académique. Un changement de variable suffit à activer le déploiement automatique.
Extrait YAML cd.yml · deploy-preprod
deploy-preprod:
  needs: build
  environment: preprod
  if: vars.PREPROD_ENABLED == 'true'

  steps:
    - name: Deploy to preprod via SSH
      uses: appleboy/ssh-action@v1
      with:
        host:     ${{ secrets.PREPROD_HOST }}
        username: ${{ secrets.PREPROD_USER }}
        key:      ${{ secrets.PREPROD_SSH_KEY }}
        script: |
          cd /opt/ressources_api
          echo ${{ secrets.GHCR_TOKEN }} | docker login \
            ghcr.io -u ${{ secrets.GHCR_USERNAME }} \
            --password-stdin
          export IMAGE_TAG=${{ needs.build.outputs.image_tag }}
          docker compose \
            -f docker-compose.yaml \
            -f docker-compose.preprod.yaml \
            pull php
          docker compose \
            -f docker-compose.yaml \
            -f docker-compose.preprod.yaml \
            up -d --no-deps php
          docker compose exec -T php \
            php bin/console doctrine:migrations:migrate \
            --no-interaction
          docker image prune -f
3
Deploy — Production
job: deploy-production
needs: build + preprod manuel seulement

Identique au déploiement préprod mais sur le serveur de production. Ne peut jamais se déclencher automatiquement — requiert un déclenchement manuel (workflow_dispatch) avec le paramètre environment: production explicitement sélectionné. Double protection : le job préprod doit aussi avoir réussi.

  1. 1needs: [build, deploy-preprod] — la prod ne peut pas être déployée sans que la préprod l'ait été avec succès. Garantit que l'image a été testée sur un environnement réel avant la prod
  2. 2Conditions cumulatives — PROD_ENABLED == 'true' ET workflow_dispatch ET inputs.environment == 'production'. Les trois conditions doivent être vraies simultanément
  3. 3environment: production — GitHub Environment séparé avec ses propres secrets (PROD_HOST, PROD_SSH_KEY…) et protection rules potentielles (reviewers obligatoires)
  4. 4Même script SSH que preprod, avec les credentials prod. Zéro downtime identique : seul le container PHP est redémarré
Triple garde-fou : (1) CI + Security doivent passer, (2) préprod doit réussir, (3) déclenchement manuel explicite avec sélection de l'environnement. Un push accidentel sur main ne peut pas atteindre la production.
Extrait YAML cd.yml · deploy-production
deploy-production:
  needs: [build, deploy-preprod]
  environment: production
  if: >
    vars.PROD_ENABLED == 'true'
    && github.event_name == 'workflow_dispatch'
    && github.event.inputs.environment == 'production'
  # 3 conditions cumulatives — jamais automatique

  steps:
    - name: Deploy to production via SSH
      uses: appleboy/ssh-action@v1
      with:
        host:     ${{ secrets.PROD_HOST }}
        username: ${{ secrets.PROD_USER }}
        key:      ${{ secrets.PROD_SSH_KEY }}
        script: |
          # Même script que preprod
          # avec docker-compose.prod.yaml
          cd /opt/ressources_api
          export IMAGE_TAG=${{ needs.build.outputs.image_tag }}
          docker compose \
            -f docker-compose.yaml \
            -f docker-compose.prod.yaml \
            pull php
          docker compose \
            -f docker-compose.yaml \
            -f docker-compose.prod.yaml \
            up -d --no-deps php
          docker compose exec -T php \
            php bin/console \
            doctrine:migrations:migrate \
            --no-interaction
