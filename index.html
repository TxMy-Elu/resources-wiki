AuthComment fonctionne l'authentification JWT ? ▾

L'authentification utilise JWT RS256 — asymétrique RSA-2048 bits. La clé privée (private.pem) signe les tokens côté serveur, la clé publique (public.pem) les vérifie. Les deux clés ne sont jamais dans le dépôt git — elles sont injectées via GitHub Secrets.

Flux complet :

  • POST /api/login → Symfony vérifie email + mot de passe (bcrypt cost 13) → retourne un access_token (TTL 3600s) et un refresh_token
  • Le client envoie Authorization: Bearer <token> à chaque requête
  • LexikJWTBundle vérifie la signature RSA et l'expiration automatiquement
  • POST /api/token/refresh → nouveau access_token sans ressaisie du mot de passe
Header:  { "alg": "RS256", "typ": "JWT" }
Payload: { "sub": "uuid", "roles": ["ROLE_CONNECTED"], "iat": 1234, "exp": 1234+3600 }
Signature: RSA-SHA256(base64(header) + "." + base64(payload), privateKey)
AuthComment sont gérés les rôles et les droits d'accès ? ▾

Le système utilise RBAC (Role-Based Access Control) avec 4 rôles hiérarchiques — chaque rôle hérite de tous les droits du précédent :

  • ROLE_CONNECTED → consulter et télécharger des ressources
  • ROLE_MODERATOR → gérer le contenu (validation, suppression)
  • ROLE_ADMIN → gérer les utilisateurs, les catégories, les stats
  • ROLE_SUPER_ADMIN → accès total, gestion des admins

Deux niveaux de contrôle coexistent dans Symfony :

  • #[IsGranted('ROLE_ADMIN')] sur le contrôleur → bloque avant l'exécution du code
  • Voters (ResourceVoter) → vérifie que l'utilisateur est propriétaire de la ressource, en plus du rôle

Le security.yaml a un catch-all : tout /api/** non listé en PUBLIC_ACCESS requiert au minimum IS_AUTHENTICATED_FULLY.

SécuComment les mots de passe sont-ils stockés ? ▾

Les mots de passe ne sont jamais stockés en clair. Symfony utilise bcrypt cost 13 via le composant PasswordHasher.

  • bcrypt : fonction de hachage adaptative. Le "cost" définit le nombre d'itérations (2¹³ = 8192). Plus le cost est élevé, plus le calcul est lent.
  • Cost 13 ≈ 100ms par hash — un attaquant avec un GPU ne peut tester qu'environ 10 tentatives/seconde, rendant la force brute non viable.
  • Chaque hash inclut un sel aléatoire unique — deux utilisateurs avec le même mot de passe ont des hashs différents (protection contre les rainbow tables).
# Symfony config/packages/security.yaml
password_hashers:
  App\Entity\Utilisateur:
    algorithm: bcrypt
    cost: 13

Le rate limiting sur POST /api/login (5 tentatives / 15 min / IP) est une protection complémentaire côté réseau.

SécuComment les injections SQL sont-elles évitées ? ▾

L'application n'écrit aucun SQL brut. Toute la persistance passe par Doctrine ORM avec des requêtes préparées (prepared statements via PDO).

  • Les paramètres utilisateur sont transmis séparément de la requête SQL — jamais concaténés dans la chaîne.
  • Le driver PDO les échappe automatiquement selon le type du paramètre.
  • Même les requêtes dynamiques (filtres, tri, pagination) utilisent setParameter() sur le QueryBuilder.
// Correct — paramètre lié
$qb->where('r.title LIKE :q')->setParameter('q', '%'.$search.'%');

// Interdit — concaténation directe (injection possible)
$qb->where("r.title LIKE '%".$search."%'"); // ← JAMAIS
CI/CDComment le déploiement est-il déclenché ? ▾

Le pipeline CI/CD est géré par GitHub Actions. Il se déclenche sur chaque push ou pull_request sur les branches main et dev.

Les jobs s'enchaînent dans l'ordre suivant :

  • 1. Tests PHPUnit — 24 tests sur une vraie base MySQL 8 (pas de mock). Si un test échoue, tout s'arrête.
  • 2. Analyse SonarCloud — qualité du code, code smells, couverture. Quality Gate configuré à 80%.
  • 3. Audit dépendances — composer audit + OWASP Dependency Check.
  • 4. Build image Docker — php:8.4-fpm-alpine avec le code Symfony.
  • 5. Scan Trivy — vulnérabilités CVE dans l'image Docker.
  • 6. Push Docker Hub — image taguée avec le SHA du commit (immuable).
  • 7. Déploiement — SSH sur le serveur, pull de la nouvelle image, redémarrage Docker Compose sans downtime.

Les secrets (clés JWT, credentials Docker Hub, SSH) sont stockés dans GitHub Secrets et injectés en variables d'environnement — jamais dans les fichiers de config.

CI/CDComment le monitoring fonctionne-t-il en production ? ▾

La stack de monitoring est composée de 4 outils déployés en conteneurs Docker dans le même réseau que l'API :

  • Prometheus — scrape les métriques exposées par l'API Symfony toutes les 15s (nombre de requêtes, temps de réponse, taux d'erreur, usage mémoire PHP).
  • Grafana — dashboards visuels en temps réel. Alertes configurées : taux d'erreur 5xx > 5%, latence p95 > 500ms, RAM > 80%.
  • Loki — base de données de logs (équivalent Elasticsearch, mais plus léger). Indexe les logs par labels (container, level, env).
  • Promtail — agent qui collecte les logs des containers Docker (/var/lib/docker/containers) et les envoie à Loki en temps réel.

Les logs applicatifs Symfony (niveaux INFO, WARNING, ERROR, CRITICAL) sont écrits dans var/log/prod.log en format JSON structuré — Promtail les parse automatiquement.

BDDQuelle est la structure de la base de données ? ▾

La base MySQL 8.0 contient les entités principales suivantes :

  • utilisateur — uuid, email, password (bcrypt), rôle, is_available (activation admin), avatar_url (Supabase)
  • ressource — uuid, titre, description, url, type, catégorie, auteur, statut (brouillon/publié/archivé)
  • categorie — arborescence simple (nom, slug)
  • participation — relation many-to-many utilisateur ↔ ressource (favoris, consultations)
  • progression — suivi de lecture par utilisateur + ressource (%, dernière position)
  • commentaire — texte, auteur, ressource, date, statut modération
  • log_action — audit trail : 16 types d'actions (créer, modifier, supprimer, modérer…), IP, timestamp
  • log_connexion — chaque tentative de login (succès ou échec), IP, user agent
  • password_reset_token — token usage unique, expires_at, used

Les identifiants sont des UUID v4 (pas d'entiers séquentiels) — un attaquant ne peut pas deviner l'ID d'une ressource appartenant à un autre utilisateur.

BDDComment les fichiers médias sont-ils gérés ? ▾

Les médias (images de ressources, avatars) ne sont pas stockés sur le serveur applicatif — ils sont externalisés dans Supabase Storage.

  • À l'upload, l'API Symfony envoie le fichier au bucket Ressources via le SDK Supabase PHP.
  • L'URL publique retournée est stockée dans la colonne url_media de l'entité.
  • La clé Supabase service_role (accès écriture) est stockée dans GitHub Secrets, jamais dans le code.
  • Seule la clé anon (lecture publique) pourrait être exposée côté client — mais l'API sert de proxy, donc même ça n'est pas nécessaire.

Ce choix évite de gérer un volume Docker pour les fichiers uploadés et simplifie la scalabilité (pas de rsync entre instances).

ArchiPourquoi PHP-FPM et pas un serveur PHP intégré ? ▾

PHP-FPM (FastCGI Process Manager) est le mode d'exécution PHP recommandé en production. Il maintient un pool de processus PHP persistants et les réutilise entre les requêtes, au lieu de démarrer un processus à chaque requête.

  • FastCGI est le protocole de communication entre Nginx et PHP-FPM (TCP sur le port :9000, réseau bridge Docker interne). Nginx ne sait pas exécuter du PHP — il délègue à PHP-FPM.
  • Isolation : Nginx (réseau public) et PHP-FPM (réseau interne Docker) sont dans des containers séparés. Un problème sur l'un n'affecte pas l'autre.
  • Performance : PHP-FPM maintient l'OPcache entre les requêtes — le code PHP est compilé en bytecode une seule fois, pas à chaque requête.
Client → Nginx :443 → (FastCGI :9000) → PHP-FPM → Symfony → MySQL
ArchiPourquoi Flutter pour le mobile et pas React Native ? ▾

Flutter a été choisi pour plusieurs raisons techniques :

  • Performances natives : Flutter compile en code natif ARM (pas de bridge JavaScript comme React Native). Les animations et les transitions sont fluidifiées à 60/120fps sans sacrifices.
  • Rendu propre : Flutter dessine sa propre UI via le moteur graphique Skia/Impeller — pas de composants natifs Android/iOS traduits. L'app a exactement le même look sur les deux plateformes.
  • Dart : langage typé statiquement, proche de Java/TypeScript, avec une courbe d'apprentissage raisonnable pour un développeur backend.
  • Distribution : un seul codebase → APK Android + IPA iOS. La release APK est buildée en CI via GitHub Actions.

L'app consomme l'API REST Symfony avec le package dio (HTTP), stocke le JWT dans le FlutterSecureStorage (keychain natif), et implémente le même RBAC que le front web.
