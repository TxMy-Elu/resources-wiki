Section 1

Plan de Déploiement

Le déploiement, c'est le processus qui fait passer le code écrit par les développeurs jusqu'aux utilisateurs finaux, de manière contrôlée, automatisée et sécurisée.

1.1 Gestion de versions — Git & Git Flow

💡 Concept — Pourquoi gérer les versions ?

Imaginez 4 développeurs qui modifient tous le même fichier en même temps, sans coordination. C'est le chaos : qui a la bonne version ? Qui a écrasé le travail de qui ? Git est un outil qui enregistre l'historique complet de chaque modification, qui l'a faite, quand, et pourquoi. Il permet de travailler en parallèle sans se marcher dessus et de revenir en arrière en cas de problème.

⚙️ Ce qu'on a mis en place

3 dépôts GitHub, un par composant :

  • ressources_api — API Symfony (back-end)
  • resources_front — Frontend Next.js
  • re_flutter — Application mobile Flutter

Branches main et dev protégées : aucun push direct autorisé. Tout changement passe par une Pull Request avec au moins 1 reviewer.

Git Flow simplifié

💡 Concept — Git Flow

Git Flow est une convention d'organisation des branches. Au lieu d'avoir tout le monde qui pousse directement sur la branche principale, chaque fonctionnalité a sa propre branche isolée. Ça évite que le travail en cours d'une personne casse le travail d'une autre.

Semantic Versioning (SemVer)

💡 Concept

SemVer est une convention de numérotation des versions : MAJEUR.MINEUR.PATCH (ex: 1.4.2). Chaque chiffre a une signification précise, permettant à n'importe qui de comprendre l'ampleur d'une mise à jour rien qu'en regardant le numéro.

v2.0.0 — MAJEUR

Changement incompatible. L'ancienne API ne fonctionne plus. Ex : refonte complète de l'authentification.

v1.1.0 — MINEUR

Nouvelle fonctionnalité rétrocompatible. L'existant continue de fonctionner.

v1.0.1 — PATCH

Correction de bug. Rien de nouveau, juste une réparation.

1.2 Intégration Continue (CI)

💡 Concept — Qu'est-ce que la CI ?

La CI (Continuous Integration) est un système automatique qui vérifie chaque modification de code dès qu'elle est poussée. Imaginez un contrôleur qualité invisible qui, à chaque fois qu'un développeur envoie du code, lance automatiquement tous les tests, vérifie le style du code et cherche des failles de sécurité. Si quelque chose ne va pas, il bloque la livraison et prévient l'équipe immédiatement — avant que le problème n'atteigne les utilisateurs.

La CI tourne sur des runners GitHub Actions : des machines virtuelles temporaires hébergées chez GitHub, qui démarrent vierges à chaque pipeline, font leur travail, et disparaissent. Aucun serveur à gérer.

⚙️ Ce qu'on a mis en place

Outil : GitHub Actions — fichiers YAML dans .github/workflows/ de chaque dépôt. Déclenché automatiquement sur chaque push ou ouverture de Pull Request.

🐘 CI — API Backend (Symfony)

Déclenchée sur chaque push ou PR vers dev ou main. Trois jobs parallèles s'enchaînent :

Job test — Tests automatisés

GitHub Actions crée une machine virtuelle avec PHP 8.4 et démarre un conteneur MySQL 8.0 en service (une base de données temporaire, juste pour ce pipeline). Puis :

  • composer install — télécharge toutes les dépendances PHP
  • Création du fichier .env.test avec l'URL de la BDD temporaire
  • php bin/console doctrine:migrations:migrate — crée les tables en BDD
  • php bin/phpunit — lance les 30 tests E2E : chaque test simule une vraie requête HTTP (login, création de ressource, modération…) et vérifie la réponse. Si un test échoue, le job échoue et bloque tout.

Job lint — Qualité du code

Vérifie que le code respecte les règles de l'équipe, sans l'exécuter :

  • PHP-CS-Fixer : vérifie que le code PHP suit le standard PSR-12 (indentation, espaces, accolades). C'est comme le "correcteur de style" — si on écrit function foo(){ au lieu de function foo() {, ça échoue.
  • symfony lint:yaml : valide que tous les fichiers YAML de config (routes, services, doctrine) sont syntaxiquement corrects.
  • symfony lint:container : vérifie que l'injection de dépendances Symfony est cohérente — pas de service déclaré mais non implémenté.

Job sonar — Analyse de sécurité SAST

Envoie l'intégralité du code source à SonarCloud pour une analyse statique approfondie :

  • Recherche de vulnérabilités connues dans les patterns de code
  • Détection de code dupliqué, fonctions trop complexes, classes trop longues
  • Vérification de la couverture de tests (% du code couvert)
  • Si le Quality Gate n'est pas passé (note en dessous du seuil défini), le job échoue et bloque le merge.
💡 Concept — Qu'est-ce qu'un test E2E ?

E2E (End-to-End = bout en bout) signifie que le test simule un vrai parcours utilisateur : il envoie une vraie requête HTTP POST /api/login, récupère le JWT dans la réponse, puis envoie une requête GET /api/ressources avec ce JWT, et vérifie que la réponse est correcte. Contrairement à un test unitaire qui teste une seule fonction isolée, l'E2E teste toute la chaîne : routeur → contrôleur → service → base de données → réponse.

⚡ CI — Frontend Web (Next.js)

Déclenchée sur chaque push ou PR vers main ou dev. Deux jobs :

Job lint — Qualité TypeScript/React

Sur un runner Node 22 avec pnpm 9 :

  • pnpm install --frozen-lockfile — installe les dépendances JS en mode strict (si le lockfile ne correspond pas au package.json, échec immédiat — ça évite les surprises de versions)
  • ESLint : analyse statique du TypeScript/React. Détecte les variables non utilisées, les imports manquants, les props incorrectes, les appels de hooks React mal placés. Règles Next.js spécifiques activées.

Job build — Build de production

Simule exactement le build qui partira en production :

  • Injection des variables d'environnement depuis les GitHub Secrets : NEXT_PUBLIC_API_URL, NEXT_PUBLIC_SUPABASE_URL, NEXT_PUBLIC_SUPABASE_ANON_KEY
  • pnpm build — compile tout le TypeScript, génère les pages SSR, optimise les images, crée les bundles JS minifiés
  • Si le build échoue (erreur TypeScript, import cassé, page qui plante), la CI échoue. C'est le même build que Vercel utilisera — si ça passe ici, ça passera en prod.

📱 CI — Application Mobile (Flutter)

Déclenchée sur chaque push ou PR vers main ou dev. Trois jobs :

Job lint — Analyse statique Dart

  • flutter pub get — télécharge les dépendances Dart/Flutter
  • flutter analyze --no-fatal-infos --no-fatal-warnings — analyse statique du code Dart. Détecte les types incorrects, les appels dépréciés, les imports inutilisés. Les warnings simples ne bloquent pas, seules les erreurs bloquent.

Job test — Tests unitaires Dart

  • flutter test --no-pub — lance tous les tests unitaires Flutter. --no-pub pour ne pas retélécharger les dépendances déjà installées par le job lint.

Job build-android — APK de production

  • Environnement Java 17 (Temurin) requis par Gradle (outil de build Android)
  • flutter build apk --release --no-pub — compile le code Dart en bytecode natif Android, crée l'APK signé
  • L'APK est archivé comme artefact GitHub Actions (rétention 7 jours). Téléchargeable depuis l'interface GitHub pour tests internes — pas encore distribué sur le Play Store.
💡 Concept — Qu'est-ce qu'un "lint" ?

Le lint analyse le code sans l'exécuter, pour trouver des erreurs de style ou de logique. C'est comme un correcteur orthographique pour le code. Si le développeur oublie une accolade, utilise une variable non définie ou ne respecte pas les conventions de l'équipe, le lint le détecte. Le nom vient de l'outil lint des années 70 qui nettoyait le code C — comme enlever les peluches (lint en anglais) d'un vêtement.

1.3 Conteneurisation Docker

💡 Concept — Qu'est-ce qu'un conteneur ?

Un conteneur, c'est comme une boîte hermétique qui contient une application et tout ce dont elle a besoin pour fonctionner (PHP, ses extensions, la configuration). Cette boîte est identique sur la machine du développeur, sur le serveur de test, et en production. Plus jamais le problème "ça marche sur mon PC" : si ça tourne dans le conteneur, ça tourne partout. Docker est la technologie qui crée et gère ces boîtes.

Différence conteneur / machine virtuelle : une VM simule un ordinateur complet avec son propre OS (lourd, lent à démarrer). Un conteneur partage le noyau Linux du serveur hôte et ne contient que ce dont l'application a besoin — il démarre en quelques secondes et pèse bien moins lourd.

💡 Concept — Réseau interne Docker (bridge)

Par défaut, les conteneurs Docker sont isolés d'Internet. Pour qu'ils se parlent entre eux, on crée un réseau interne appelé bridge. Sur ce réseau, chaque conteneur a un nom qui sert d'adresse (ex: resources_db est joignable par PHP à l'adresse resources_db:3306). Depuis l'extérieur (Internet), seuls les ports qu'on a explicitement ouverts sont accessibles. C'est un pare-feu automatique.

Les 5 conteneurs et leur rôle détaillé

🌐 resources_nginx — nginx:alpine | Port 8080→80

Rôle : Reverse proxy + terminaison SSL. C'est la porte d'entrée unique de toute l'infrastructure.

💡 Qu'est-ce qu'un reverse proxy ?

Un proxy normal, c'est un intermédiaire côté client (ex: proxy d'entreprise qui filtre le trafic sortant). Un reverse proxy est l'inverse : il se place côté serveur et reçoit toutes les requêtes des clients à la place des vrais serveurs. Les clients ne connaissent que l'adresse du reverse proxy — jamais l'adresse de PHP ou de MySQL. Nginx reçoit la requête HTTP, regarde l'URL, et décide où l'envoyer : si c'est /api/*, il transfère à PHP-FPM. Si c'est un fichier statique, il le sert lui-même directement (beaucoup plus rapide que PHP).

  • Terminaison SSL : Nginx gère le chiffrement HTTPS. PHP reçoit les requêtes en HTTP simple en interne — plus rapide et plus simple. Nginx s'occupe de déchiffrer/chiffrer avec le certificat Let's Encrypt.
  • Gestion des fichiers statiques : images, CSS, JS du front Symfony sont servis directement par Nginx sans passer par PHP (Nginx est 10× plus rapide pour ça).
  • Buffering des requêtes : Nginx accumule les requêtes lentes des clients et les transmet à PHP d'un coup — PHP-FPM n'est pas bloqué par les connexions lentes.
  • Image alpine : version ultra-légère de Linux — l'image ne pèse que ~25 MB contre ~150 MB pour l'image standard.

🐘 resources_php — php:8.4-fpm | Port interne uniquement

Rôle : Runtime PHP — exécute le code Symfony (l'API REST).

💡 Qu'est-ce que PHP-FPM ?

PHP peut fonctionner de plusieurs façons. En mode CLI, PHP exécute un script et s'arrête. En mode FPM (FastCGI Process Manager), PHP tourne en permanence en arrière-plan avec un pool de workers prêts à traiter des requêtes. Quand Nginx reçoit une requête pour /api/login, il la passe à PHP-FPM via le protocole FastCGI. Un worker PHP-FPM disponible la prend, exécute le contrôleur Symfony, et renvoie la réponse à Nginx. Avec FPM, on peut configurer combien de workers PHP tournent simultanément, évitant de surcharger le serveur.

  • Extensions installées : pdo_mysql (connexion MySQL), intl (internationalisation), opcache (cache du bytecode PHP — ne recompile pas le PHP à chaque requête)
  • Utilisateur non-root : PHP tourne en tant qu'utilisateur symfony (UID 1001). Si quelqu'un exploite une faille dans l'API, il n'obtient pas les droits root sur le serveur.
  • HEALTHCHECK : Docker teste régulièrement que PHP-FPM répond (php-fpm -t). Si ce n'est plus le cas, Docker redémarre automatiquement le conteneur.

🗄️ resources_db — mysql:8.0 | Port 3307→3306 (dev uniquement)

Rôle : Base de données relationnelle — stocke toutes les données de l'application.

💡 Qu'est-ce qu'une base de données relationnelle ?

MySQL est un SGBDR (Système de Gestion de Base de Données Relationnelle). Les données sont organisées en tables (comme des feuilles Excel) reliées entre elles par des clés étrangères. Par exemple : une table users, une table ressources, une table categories. Doctrine ORM (intégré à Symfony) traduit les objets PHP en requêtes SQL automatiquement — les développeurs manipulent des objets PHP, pas du SQL brut.

  • Le port 3306 est exposé en dev (3307→3306) pour permettre la connexion depuis un outil comme DBeaver ou TablePlus. En production, aucun port MySQL n'est exposé — la BDD n'est accessible que depuis le réseau bridge interne Docker.
  • Volume persistant : les données MySQL sont stockées dans un volume Docker nommé. Si le conteneur est supprimé et recréé, les données survivent.
  • 3 bases distinctes : resources (dev local), resources_preprod (préprod), resources_prod (production).

🖥️ resources_pma — phpmyadmin | Port 8081→80 | Dev uniquement

Rôle : Interface web pour administrer MySQL visuellement.

💡 Qu'est-ce que phpMyAdmin ?

phpMyAdmin est une application web qui permet de naviguer dans les tables MySQL, d'exécuter des requêtes SQL, d'importer/exporter des données, sans écrire de commande en ligne. C'est un outil de commodité pour les développeurs : au lieu de se connecter en SSH et de taper mysql -u root -p, on ouvre localhost:8081 dans son navigateur. En production, ce conteneur n'existe pas — trop risqué d'exposer une interface d'admin.

  • Accessible uniquement sur la machine du développeur, jamais exposé vers l'extérieur.
  • Absent des environnements préprod et production (docker-compose.override.yaml uniquement).

📧 resources_mail — axllent/mailpit | Port 8025 | Dev uniquement

Rôle : Faux serveur SMTP — intercepte tous les emails envoyés par l'application en dev.

💡 Pourquoi un faux serveur mail ?

L'application envoie des emails réels : email d'activation de compte, reset de mot de passe, notifications. En développement, on ne veut surtout pas envoyer de vrais emails à de vraies adresses (risque de spammer des gens, d'exposer des données, et la configuration SMTP réelle n'est pas disponible). Mailpit fait semblant d'être un serveur SMTP : Symfony lui envoie les emails, et ils sont capturés et affichés dans une interface web sur localhost:8025. Le développeur peut voir exactement ce que l'email contiendrait, sans qu'il parte vraiment.

  • Interface web intégrée pour visualiser les emails capturés (localhost:8025).
  • En production, le vrai SMTP Gmail est configuré via les variables d'environnement.

Dockerfile multi-stage — pourquoi deux étapes ?

💡 Concept — Dockerfile multi-stage

Un build multi-stage, c'est comme construire une voiture dans une usine en plusieurs étapes, mais en ne livrant au client que la voiture finie — pas l'atelier, les outils, et les pièces de rechange. On installe les dépendances dans un premier conteneur "de construction" (qui a composer, des outils de build…), puis on ne copie que le résultat final dans l'image légère qui part en production. Résultat : l'image de production est bien plus petite (pas de composer, pas de dépendances de dev) et sa surface d'attaque est réduite.

Stage 1 — "builder" (image composer:2)
  └─ composer install --no-dev --optimize-autoloader
     ↳ Installe uniquement les dépendances de PRODUCTION (pas les outils de test)
     ↳ --optimize-autoloader : génère une map de classes pour accélérer l'autoloading

Stage 2 — "runtime" (image php:8.4-fpm-alpine — ~50 MB)
  ├─ Installe uniquement les extensions PHP nécessaires : pdo_mysql, intl, opcache
  ├─ Crée l'utilisateur non-root : symfony (UID 1001)
  ├─ Copie uniquement /vendor/ depuis Stage 1 (pas composer lui-même)
  ├─ Copie le code source de l'application
  └─ HEALTHCHECK : docker vérifie que php-fpm répond toutes les 30s
⚙️ Points clés sécurité conteneurs
  • Utilisateur non-root : PHP tourne en tant qu'utilisateur symfony (UID 1001) — si l'API est compromise, l'attaquant n'a pas les droits root sur le serveur hôte.
  • Réseau bridge isolé : tous les conteneurs communiquent via symfony_network. Seul Nginx expose des ports vers l'extérieur. MySQL et PHP-FPM sont totalement invisibles depuis Internet.
  • Images Alpine : basées sur Alpine Linux (~5 MB) au lieu de Debian/Ubuntu (~150 MB) — moins de packages installés = moins de vulnérabilités potentielles.
  • HEALTHCHECK automatique : Docker surveille les conteneurs et les redémarre si nécessaire, sans intervention humaine.

1.4 Déploiement Continu (CD)

💡 Concept — CI vs CD

CI = vérifier que le code est correct (tests, lint, sécurité). CD = automatiser la livraison de ce code vérifié jusqu'aux serveurs. Le CD s'enchaîne après la CI : si tous les contrôles sont au vert, le code est automatiquement emballé et expédié vers l'environnement cible. Pour la production, une approbation manuelle est obligatoire — un humain doit valider avant que la mise en prod se lance.

1.6 Gestion des secrets

💡 Concept — Pourquoi ne jamais mettre de secrets dans le code ?

Un secret dans le code, c'est comme écrire votre code de carte bleue sur le ticket de caisse et le laisser traîner. Le code source est souvent partagé, versionné, et parfois public. Un mot de passe de base de données dans un fichier .env commité sur GitHub peut être trouvé par des robots en quelques minutes. GitHub Secrets stocke ces valeurs de façon chiffrée et ne les injecte que dans les pipelines CI/CD au moment de l'exécution — elles n'apparaissent jamais dans les logs.

🔒 Règle absolue du projet

Les fichiers .env*, config/jwt/ et generate_keys.php sont dans le .gitignore et ne sont JAMAIS commités. Les secrets viennent uniquement des GitHub Secrets. Un scan Gitleaks tourne en CI pour détecter tout secret accidentellement introduit dans l'historique git.

Section 2

Plan de Sécurisation

La sécurité n'est pas une fonctionnalité ajoutée à la fin — c'est une démarche intégrée dès la conception. Cette section détaille toutes les mesures mises en place pour protéger l'application, les données et les utilisateurs.

2.1 Authentification — JWT RS256

💡 Concept — Qu'est-ce qu'un JWT ?

Un JWT (JSON Web Token) est comme un badge d'accès signé. Quand vous vous connectez, le serveur vous donne un badge contenant votre identité et vos droits, signé cryptographiquement. À chaque requête suivante, vous présentez ce badge. Le serveur vérifie la signature sans aller interroger la base de données — c'est plus rapide et "stateless" (sans état côté serveur).

💡 Concept — RS256 vs HS256 : pourquoi des clés asymétriques ?

HS256 utilise la même clé pour signer et vérifier (comme une serrure dont tout le monde a la même clé). RS256 utilise une paire de clés : la clé privée signe (seul le serveur l'a), la clé publique vérifie (peut être partagée). Si un attaquant vole la clé publique, il ne peut pas forger des tokens. C'est nettement plus sécurisé dans un système distribué (web + mobile).

⚙️ Ce qu'on a mis en place
  • Bundle : LexikJWTBundle (Symfony)
  • Algorithme : RSA-2048 bits, signature RS256
  • TTL : 3600 secondes (1 heure)
  • Clés : fichiers PEM dans config/jwt/, exclus du git, stockés en GitHub Secrets (base64)
  • Reset mot de passe : token unique, usage unique, validité 1 heure
  • Compte inactif : is_available = false jusqu'à activation admin

2.2 OWASP Top 10

💡 Qu'est-ce que l'OWASP Top 10 ?

L'OWASP (Open Web Application Security Project) est une fondation à but non lucratif qui produit des ressources libres sur la sécurité des applications web. Son document le plus connu, le Top 10, liste les 10 catégories de vulnérabilités les plus critiques et les plus répandues. Il est mis à jour tous les 3-4 ans à partir de données réelles de milliers d'applications auditées. C'est devenu la référence mondiale : les certifications (OSCP, CEH), les audits de sécurité et les cahiers des charges exigent systématiquement qu'une application y réponde. Notre bilan : 8/10 couverts, 2/10 partiels.

✅ 8/10 Couverts

⚠️ 2/10 Partiels

A01 — Contrôle d'accès défaillant  Couvert

💡 C'est quoi ?

Un contrôle d'accès, c'est la règle qui décide qui a le droit de faire quoi. Une défaillance de contrôle d'accès signifie qu'un utilisateur peut accéder à des données ou des actions auxquelles il ne devrait pas avoir accès. C'est la vulnérabilité numéro 1 depuis 2021. Exemples concrets : un utilisateur normal qui accède à /api/admin/users et voit tous les comptes, ou qui modifie la ressource d'un autre utilisateur en changeant l'ID dans l'URL (/api/ressources/42 au lieu de /api/ressources/17). C'est souvent simplement parce que le développeur a oublié de vérifier le droit d'accès côté serveur (en pensant que le front-end "cachait" le bouton).

Ce qu'on a mis en place :

  • RBAC (Role-Based Access Control) — 4 rôles : ROLE_CONNECTED, ROLE_MODERATOR, ROLE_ADMIN, ROLE_SUPER_ADMIN. Chaque rôle hérite des précédents (hiérarchie Symfony). Un admin a automatiquement tous les droits d'un modérateur.
  • Voters Symfony : pour les ressources, le contrôle ne s'arrête pas au rôle — on vérifie aussi que l'utilisateur est bien le propriétaire de la ressource (ResourceVoter::canEdit()). Un utilisateur lambda ne peut pas modifier la ressource d'un autre même s'il connaît son ID.
  • #[IsGranted] sur chaque contrôleur : chaque endpoint a son attribut de contrôle. Si quelqu'un essaie d'accéder à /api/admin/users sans être admin, Symfony retourne 403 automatiquement, avant même d'exécuter le code métier.
  • security.yaml — catch-all : tout /api non explicitement listé comme PUBLIC_ACCESS requiert au minimum ROLE_CONNECTED. Une route oubliée est bloquée par défaut, pas ouverte.

A02 — Défaillances cryptographiques  Couvert

💡 C'est quoi ?

Cette catégorie couvre tous les cas où des données sensibles transitent ou sont stockées sans chiffrement suffisant, ou avec un chiffrement trop faible. Exemples : stocker les mots de passe en clair (ou en MD5) dans la base de données, transmettre des données en HTTP (non HTTPS), utiliser un vieux protocole TLS 1.0 cassé, ou utiliser un token d'authentification non signé. C'est souvent la porte d'entrée vers des fuites de données massives — quand la base de données est volée, les mots de passe doivent être illisibles.

Ce qu'on a mis en place :

  • TLS 1.2+ : toutes les communications client-serveur passent par HTTPS. TLS (Transport Layer Security) chiffre le flux réseau — même si quelqu'un intercepte les paquets sur le réseau, il ne voit que du bruit. Le certificat SSL est géré par Nginx + Let's Encrypt (renouvellement automatique).
  • bcrypt cost 13 : les mots de passe ne sont jamais stockés en clair. bcrypt applique une fonction de hachage avec "sel" (une chaîne aléatoire unique par mot de passe, pour éviter les attaques par rainbow table). Le cost 13 signifie 2¹³ = 8192 itérations — chaque vérification prend ~100ms, rendant la force brute non viable (10 tentatives/s max). MD5 : des milliards de tentatives/seconde.
  • JWT RS256 : les tokens sont signés avec RSA-2048 bits. Ni stockés dans la base de données, ni transmis via URL (uniquement dans le header Authorization: Bearer).
  • Clés privées hors du code : les clés PEM RSA ne sont jamais dans le dépôt git. Elles sont stockées dans GitHub Secrets et injectées en variable d'environnement au déploiement.

A03 — Injection  Couvert

💡 C'est quoi ?

Une injection, c'est quand un attaquant envoie des données qui vont être interprétées comme des commandes par le système. L'exemple classique : l'injection SQL. Si une application construit sa requête SQL en concaténant directement ce que l'utilisateur tape, un attaquant peut "injecter" du SQL. Exemple : si le login fait "SELECT * FROM users WHERE email = '" + email + "'" et qu'on envoie ' OR '1'='1 comme email, la requête devient SELECT * FROM users WHERE email = '' OR '1'='1' — qui retourne tous les utilisateurs. L'attaquant peut se connecter sans mot de passe, voire supprimer toute la base.

Ce qu'on a mis en place :

  • Doctrine ORM : on n'écrit jamais de SQL brut. Doctrine traduit les objets PHP en requêtes SQL en utilisant des requêtes préparées (prepared statements). Les paramètres utilisateur sont transmis séparément de la requête — jamais concaténés. Le driver PDO les échappe automatiquement. Un attaquant ne peut pas "sortir" de la requête.
  • QueryBuilder Doctrine : même quand on construit des requêtes dynamiques (filtres, recherches), on utilise ->setParameter() — jamais d'interpolation de chaîne directe.
  • Validation des entrées : le Validator Symfony vérifie le format des données avant qu'elles atteignent la couche de persistance (email valide, longueurs respectées, types corrects).

A04 — Conception non sécurisée  Couvert

💡 C'est quoi ?

Cette catégorie, introduite en 2021, vise les failles qui viennent non pas d'un bug de code, mais d'un mauvais design de l'application. Par exemple : permettre à n'importe qui de créer un compte admin, ne pas limiter le nombre de tentatives de connexion (brute force), envoyer le token de reset de mot de passe par email sans expiration, ou créer une API où on peut deviner les IDs des autres utilisateurs (ID séquentiel). Ce sont des erreurs de conception qui ouvrent des brèches même si le code est techniquement correct.

Ce qu'on a mis en place :

  • Activation des comptes par un admin : un nouvel utilisateur ne peut pas se connecter immédiatement après l'inscription. Un administrateur doit activer manuellement le compte (is_available = false par défaut). Cela empêche la création automatisée de comptes malveillants.
  • Rate limiting (limitation du débit) : le composant Symfony Rate Limiter bloque automatiquement une IP après 5 tentatives de login en 15 minutes. Même chose pour l'inscription (10/heure) et le reset de mot de passe (3/heure). Même si l'attaquant connaît l'email, il ne peut pas tester 10 000 mots de passe.
  • Token de reset usage unique : le token envoyé par email pour réinitialiser un mot de passe est à usage unique (used = false → true après utilisation) et expire après 1 heure. On ne peut pas le réutiliser.
  • UUID comme identifiants : les ressources utilisent des UUIDs (identifiants aléatoires) plutôt que des entiers séquentiels (1, 2, 3…). Un attaquant ne peut pas deviner l'ID d'une ressource appartenant à un autre utilisateur.

A05 — Mauvaise configuration de sécurité  Partiel

💡 C'est quoi ?

Cette catégorie couvre toutes les erreurs de configuration du serveur, du framework ou de l'application qui créent des failles. Exemples : laisser le mode debug activé en production (qui affiche le code source et les variables internes), exposer des ports inutiles, ne pas configurer les headers de sécurité HTTP, laisser les comptes par défaut (admin/admin), ou avoir un CORS trop permissif (Access-Control-Allow-Origin: * qui autorise n'importe quel site à faire des requêtes). Les headers HTTP de sécurité sont des instructions que le serveur envoie au navigateur pour le "verrouiller".

Ce qu'on a fait :

  • APP_ENV=prod en production : le mode debug est désactivé. Symfony n'affiche pas les erreurs détaillées aux utilisateurs — seulement un message générique.
  • CORS configuré (NelmioCorsBundle) : seuls les domaines autorisés (Vercel front, app mobile) peuvent faire des requêtes cross-origin à l'API. Access-Control-Allow-Origin n'est pas *.
  • Ports fermés : seul le port 443 (HTTPS) est ouvert vers l'extérieur. MySQL, PHP-FPM, et les outils de monitoring sont uniquement accessibles depuis le réseau bridge Docker interne.
⚠️ Ce qui manque — CSP (Content Security Policy)

Un header CSP est une instruction que le serveur envoie au navigateur pour lui dire d'où il est autorisé à charger du JavaScript, des images, des styles. Sans CSP, si un attaquant réussit à injecter du HTML dans la page (XSS), le navigateur exécutera n'importe quel script, même chargé depuis un domaine externe malveillant. Avec un CSP, le navigateur refusera ce script. À implémenter dans next.config.js :

headers: [{ key: 'Content-Security-Policy',
  value: "default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline';" }]

A06 — Composants vulnérables et obsolètes  Couvert

💡 C'est quoi ?

Une application moderne utilise des dizaines de bibliothèques tierces (dépendances). Chacune peut avoir des vulnérabilités découvertes après coup. Si on ne met pas à jour les dépendances, on peut se retrouver avec une faille critique connue publiquement (CVE — Common Vulnerabilities and Exposures) qu'un attaquant peut exploiter. Log4Shell (2021) est l'exemple parfait : une faille dans la bibliothèque Java log4j a touché des millions de serveurs dans le monde. Les entreprises qui avaient fait leur audit de dépendances l'ont corrigée rapidement. Les autres ont été compromises.

Ce qu'on a mis en place :

  • Dependabot (GitHub) : bot automatique configuré sur chaque dépôt. Dès qu'une dépendance a une mise à jour ou une CVE connue, Dependabot ouvre automatiquement une Pull Request. L'équipe n'a plus à surveiller manuellement les mises à jour.
  • composer audit : commande Composer qui interroge la base de données des CVE PHP et signale si une dépendance installée a une vulnérabilité connue. Lancé en CI à chaque push.
  • pnpm audit : équivalent pour les dépendances JavaScript (Next.js, bibliothèques React). Lance l'audit de la base npm security advisory.
  • Trivy : scanner de vulnérabilités qui analyse l'image Docker (php:8.4-fpm-alpine) et remonte les CVE dans les paquets système Alpine. Intégré en CI sur le job de build Docker.
  • SonarCloud : détecte aussi les dépendances connues vulnérables via son analyse statique.

A07 — Identification et authentification de mauvaise qualité  Couvert

💡 C'est quoi ?

Cette catégorie couvre les failles dans le processus d'authentification lui-même : sessions qui ne s'invalidement pas à la déconnexion, tokens qui ne expirent jamais, mots de passe faibles acceptés, possibilité de réutiliser un token de reset, ou absence de protection contre la force brute. Concrètement : si un token JWT n'expire jamais et qu'il est volé (via une fuite de logs, une XSS, un ordinateur partagé), l'attaquant a un accès permanent. C'est pour ça que les tokens doivent expirer.

Ce qu'on a mis en place :

  • JWT TTL 3600s (1 heure) : les tokens expirent automatiquement après 1 heure. Passé ce délai, le client doit se reconnecter ou utiliser le refresh token. Si un token est volé, la fenêtre d'exploitation est limitée à 1 heure maximum.
  • Refresh token : plutôt que d'obliger l'utilisateur à retaper son mot de passe toutes les heures, le système propose un refresh token (durée plus longue) pour obtenir un nouveau JWT sans saisie de mot de passe. Le refresh token est stocké côté serveur et peut être révoqué.
  • Token de reset usage unique : le token envoyé par email pour réinitialiser le mot de passe est invalidé dès son utilisation (colonne used dans la table password_reset_token). Il ne peut pas être utilisé deux fois.
  • RS256 asymétrique : un token JWT ne peut pas être forgé sans la clé privée (connue uniquement du serveur). Même si quelqu'un décode le JWT (c'est encodé en base64, pas chiffré), il ne peut pas modifier le payload et recalculer une signature valide.
  • Rate limiting sur le login : 5 tentatives par 15 minutes par IP — protection contre la force brute sur les mots de passe.

A08 — Manque d'intégrité des données et du logiciel  Partiel

💡 C'est quoi ?

Cette catégorie couvre les cas où on fait confiance à du code ou des données sans vérifier qu'ils n'ont pas été altérés. L'exemple le plus courant : les mises à jour automatiques. Si une application télécharge et installe automatiquement une mise à jour sans vérifier sa signature cryptographique, un attaquant qui compromet le serveur de distribution peut faire installer un logiciel malveillant. Même chose pour les releases GitHub : si personne ne peut vérifier qu'une release v1.2.0 est authentique (publiée par l'équipe), un attaquant qui compromet le compte GitHub peut publier une release modifiée.

Ce qu'on a fait :

  • composer.lock versionné : le fichier composer.lock fixe exactement la version de chaque dépendance PHP (pas seulement la contrainte, mais le hash exact du paquet). En CI, composer install installe exactement ces versions — impossible d'installer silencieusement une version différente.
  • pnpm-lock.yaml versionné : même principe pour les dépendances JavaScript.
  • Images Docker taguées par SHA : en CI, on référence les images Docker par leur hash SHA256 (immuable) plutôt que par un tag :latest qui peut pointer vers des versions différentes selon le moment.
⚠️ Ce qui manque — Signature GPG des releases

Les releases GitHub ne sont pas signées cryptographiquement. Signer une release, c'est apposer une "empreinte digitale" cryptographique qui prouve que c'est bien l'équipe qui l'a publiée. À implémenter : git tag -s v1.2.0 -m "Release 1.2.0" (l'option -s signe avec GPG). N'importe qui peut alors vérifier l'authenticité avec git tag -v v1.2.0. Dans notre cas le risque est faible (pas de distribution automatique aux utilisateurs), mais c'est une bonne pratique à adopter.

A09 — Carence en journalisation et surveillance  Couvert

💡 C'est quoi ?

Si quelqu'un s'introduit dans votre application, comment le savez-vous ? Si vous n'avez pas de logs, vous ne le saurez pas — ou trop tard. La journalisation (logging) consiste à enregistrer tous les événements significatifs : connexions, tentatives échouées, modifications de données, erreurs. La surveillance (monitoring) consiste à analyser ces logs en temps réel et à alerter quand quelque chose d'anormal se produit. Sans ça, un attaquant peut passer des semaines dans le système sans être détecté. La règle des 72h de notification RGPD (en cas de fuite de données) est impossible à respecter si on n'a pas de logs pour savoir ce qui s'est passé.

Ce qu'on a mis en place :

  • Table log_action : 16 types d'actions loguées en base de données : création/modification/suppression de ressource, changement de rôle, modération, etc. Chaque log contient : utilisateur, action, entité concernée, IP, timestamp.
  • Table log_connexion : chaque tentative de connexion (réussie ou non) est loguée avec l'IP et le timestamp. Permet de détecter une attaque par force brute même si le rate limiter l'a bloquée.
  • Table log_systeme : logs des événements système (démarrage, erreurs critiques, jobs planifiés).
  • Stack Loki + Promtail : les logs applicatifs Symfony (fichiers var/log/) et les logs Docker sont collectés par Promtail et stockés dans Loki (base de données de logs). Grafana permet de les consulter, de filtrer et de créer des alertes en temps réel.
  • Prometheus + Grafana : métriques en temps réel — nombre de requêtes par seconde, temps de réponse moyen, taux d'erreurs 4xx/5xx, usage CPU/RAM. Une alerte se déclenche si le taux d'erreur dépasse un seuil.

A10 — Falsification de requête côté serveur (SSRF)  Couvert

💡 C'est quoi ?

SSRF (Server-Side Request Forgery) est une attaque où un attaquant fait faire au serveur des requêtes HTTP que lui-même ne pourrait pas faire. Exemple : une API qui permet d'entrer une URL pour "prévisualiser un lien". Si l'API fait une requête HTTP vers cette URL sans restriction, l'attaquant peut lui faire appeler http://169.254.169.254/latest/meta-data/ (l'API de métadonnées des serveurs AWS) et récupérer les clés d'accès cloud. Ou cibler des services internes (http://localhost:6379 pour Redis, http://database:3306 pour MySQL) auxquels seul le serveur a accès. C'est souvent utilisé pour "pivoter" depuis un service exposé vers des services internes protégés.

Ce qu'on a mis en place :

  • Aucun appel HTTP initié par des entrées utilisateur : l'API Symfony ne fait aucune requête HTTP vers des URLs fournies par l'utilisateur. Il n'y a pas de fonctionnalité de "preview de lien", de "webhook URL", ni de "fetch d'image distante à partir d'une URL". La surface d'attaque SSRF est donc nulle.
  • Les seuls appels HTTP sortants : l'envoi d'emails (SMTP) et les appels Supabase pour le stockage de fichiers. Ces URLs sont fixées dans la configuration serveur, jamais contrôlées par l'utilisateur.

2.6 Chiffrement des données

💡 Concept — En transit vs Au repos

En transit = les données pendant qu'elles voyagent sur le réseau (entre le navigateur et le serveur). Sans chiffrement, n'importe qui sur le même réseau peut lire votre mot de passe. 💡 Concept — Qu'est-ce que bcrypt ?

bcrypt est un algorithme de hachage volontairement lent. Contrairement à MD5 ou SHA-1, il est conçu pour prendre du temps (le "cost" ajuste la lenteur). Sur un GPU moderne, craquer 1 milliard de mots de passe MD5/seconde est possible. Avec bcrypt cost 13, le même GPU ne peut en tester que quelques milliers. Le hachage est aussi "salé" : deux utilisateurs avec le même mot de passe auront des hachés différents.

💡 Concept — RPO vs RTO

Dossier de Maintenance

La maintenance est ce qui se passe après la livraison. Un produit non maintenu vieillit mal : les dépendances deviennent vulnérables, les fonctionnalités évoluent, les bugs s'accumulent. Ce dossier formalise comment on gère tout ça.

3.1 Types de maintenance

🛡️ Préventive — Anticiper

Actions régulières pour éviter les problèmes avant qu'ils surviennent. Mise à jour des dépendances, rotation des clés, vérification des logs, sauvegarde BDD. C'est comme la révision régulière d'une voiture.

🚀 Évolutive — Faire évoluer

Nouvelles fonctionnalités, intégrations tierces, refontes partielles. Chaque évolution est estimée en jours et tarifiée (TJM : 350 €HT/j). Garantie 30 jours contre régressions.

🔧 Corrective — Réparer

Correction des anomalies remontées. 4 niveaux de sévérité avec SLA différents : de 2h (critique) à 3 jours ouvrés (cosmétique).

3.2 Outil de gestion — GitHub Issues

💡 Concept — Pourquoi un outil de gestion de tickets ?

Un bug signalé par email ou par oral, ça se perd. Un outil de ticketing centralise toutes les demandes, les priorise, les assigne, suit leur progression et conserve l'historique. GitHub Issues est intégré directement au dépôt de code : un ticket peut référencer un commit (fixes #33 dans un message de commit ferme automatiquement l'issue), une PR peut être liée à une issue. Pas besoin d'un outil tiers séparé.

Déploiement & Infrastructure
Pourquoi utiliser Git Flow plutôt que de pousser directement sur main ?

Pousser directement sur main présente plusieurs risques : du code non testé peut partir en production, deux développeurs peuvent créer des conflits impossibles à résoudre, et il est impossible de revenir en arrière proprement. Git Flow impose que tout travail passe par une branche dédiée et une Pull Request avec review humaine + CI verte avant d'être intégré. Ça garantit que rien de cassé n'atteint la production.

Quelle est la différence entre la CI et le CD ?

La CI (Intégration Continue) vérifie automatiquement que le code est correct : elle lance les tests, le lint, les audits de sécurité et le build à chaque push. C'est la phase de validation. Le CD (Déploiement Continu) prend le code validé par la CI et l'envoie automatiquement vers les serveurs. C'est la phase de livraison. Les deux s'enchaînent : CI d'abord, CD seulement si tout est vert.

Pourquoi Docker plutôt qu'une installation directe sur le serveur ?

Une installation directe sur le serveur crée des problèmes : "ça marche sur ma machine" mais pas sur le serveur à cause de différences de versions PHP ou d'extensions manquantes. Docker encapsule l'application et tout son environnement dans une image identique partout. De plus, Docker permet de lancer plusieurs versions simultanément (préprod + prod sur le même serveur), d'isoler les services via un réseau interne, et de revenir à une version précédente en changeant simplement le tag de l'image.

Pourquoi avoir besoin d'une approbation manuelle pour la mise en production ?

Même si la CI est verte, il peut y avoir des raisons humaines de ne pas déployer : un autre déploiement est en cours, c'est vendredi soir avant un week-end, un autre bug critique vient d'être découvert, ou on attend de grouper plusieurs changements. L'approbation manuelle via GitHub Environments crée un point de contrôle humain dans un processus par ailleurs entièrement automatisé. Le "qui" approuve est tracé dans l'historique GitHub.

Que se passe-t-il si la CI échoue ?

Le pipeline s'arrête à l'étape en échec et tout le reste est bloqué. Le merge de la PR est verrouillé (GitHub bloque le bouton merge si des checks sont rouges). Le développeur reçoit une notification. Les étapes sont bloquantes dans l'ordre : si les tests échouent, le build Docker ne se lance pas. Si le build échoue, le déploiement ne se lance pas. C'est un filet de sécurité automatique.

À quoi sert Vercel pour une preview par PR ?

Pour chaque Pull Request, Vercel déploie automatiquement le frontend sur une URL unique (ex: resources-front-pr-42.vercel.app). Ça permet à n'importe quel membre de l'équipe — y compris non-technique — de voir visuellement les changements sans installer quoi que ce soit. Le lien est posté automatiquement dans la PR. Une fois la PR mergée, l'URL de preview est supprimée.

Sécurité
Pourquoi RS256 plutôt que HS256 pour les JWT ?

HS256 utilise un secret partagé : quiconque peut vérifier un token peut aussi en forger un. RS256 utilise une paire de clés asymétriques : seul le serveur a la clé privée (pour signer), la clé publique peut être distribuée (pour vérifier uniquement). Dans un système distribué avec une application web et mobile, si la clé publique est exposée, l'attaquant ne peut toujours pas créer de faux tokens. De plus, si une clé est compromise, seule la clé privée est sensible.

Comment fonctionne le rate limiting dans votre application ?

Le rate limiting est implémenté au niveau de l'API Symfony. Pour l'endpoint de login : maximum 5 tentatives par tranche de 15 minutes par IP. Pour le reset de mot de passe : maximum 3 demandes par heure. Si la limite est atteinte, l'API répond avec un code 429 (Too Many Requests). Ce mécanisme protège contre les attaques brute force : à 5 essais toutes les 15 minutes, il faudrait des milliers d'années pour tester toutes les combinaisons d'un mot de passe complexe.

Qu'est-ce qu'une injection SQL et comment vous protégez-vous ?

Une injection SQL, c'est quand un utilisateur malveillant insère du code SQL dans un champ de formulaire pour manipuler la base de données. Exemple : si on construit "SELECT * FROM users WHERE name = '" + input + "'" et que l'input est ' OR 1=1 --, on récupère tous les utilisateurs. La protection : on utilise exclusivement Doctrine ORM avec des paramètres liés. Doctrine envoie la requête et les données séparément au moteur SQL — les données ne peuvent jamais être interprétées comme du code SQL.

Pourquoi les mots de passe sont hashés avec bcrypt et pas MD5 ou SHA ?

MD5 et SHA sont des algorithmes rapides, conçus pour être calculés le plus vite possible. Sur un GPU moderne, on peut calculer des milliards de hachages MD5 par seconde. bcrypt est volontairement lent : le paramètre "cost 13" signifie qu'il faut environ 300ms pour hasher un mot de passe. Un attaquant qui vole la base de données ne peut tester que quelques milliers de combinaisons par seconde, contre des milliards avec MD5. De plus, bcrypt "sale" automatiquement chaque hash, rendant les tables pré-calculées (rainbow tables) inutilisables.

Qu'est-ce que Gitleaks et pourquoi en avez-vous besoin ?

Gitleaks est un outil qui scanne l'historique complet d'un dépôt git à la recherche de secrets accidentellement commités : mots de passe, clés API, tokens, chaînes de connexion BDD. Le problème : même si on supprime un fichier dans le dernier commit, il reste dans l'historique git et quelqu'un peut le retrouver. Gitleaks tourne en CI à chaque push et bloque si un pattern suspect est détecté (ex: une chaîne qui ressemble à une clé AWS, un token JWT, un mot de passe). On a aussi configuré un fichier .gitleaks.toml pour les faux positifs.

Quels sont les 2 points OWASP non encore complètement traités et pourquoi ?

A05 — Security Misconfiguration : les headers HTTP de sécurité comme Content-Security-Policy (CSP) et X-Frame-Options ne sont pas encore configurés. CSP est complexe à mettre en place (il faut lister toutes les sources autorisées pour les scripts, images, fonts) et peut casser des fonctionnalités si mal configuré. C'est en feuille de route.

A08 — Software and Data Integrity : le composer.lock est versionné (bonne pratique), mais les releases GitHub ne sont pas encore signées cryptographiquement. La signature permettrait de vérifier qu'une release n'a pas été modifiée entre sa création et son déploiement.

Comment fonctionnent concrètement vos logs de sécurité ?

L'application maintient 4 tables de logs en base de données. log_connexion enregistre chaque tentative de login (succès ou échec) avec l'IP, l'horodatage et le résultat. log_action enregistre chaque action métier sensible (create, save, moderate, delete…) avec l'utilisateur qui l'a faite. Ces logs permettent de répondre à des questions comme "qui a supprimé cette ressource ?" ou "depuis quelles IPs ce compte a-t-il été utilisé ?". La rétention est de 12 mois pour respecter le RGPD (intérêt légitime de sécurité).

Pourquoi la BDD MySQL n'est-elle pas chiffrée au repos ?

C'est un choix conscient et assumé, documenté dans le plan de sécurisation. Le volume Docker MySQL n'est pas chiffré nativement, mais il est protégé par plusieurs autres couches : il est isolé sur un réseau bridge privé (aucun port 3306 exposé vers l'extérieur en production), l'accès SSH au serveur est protégé par clé, et les mots de passe en base sont hachés bcrypt (une fuite de la BDD ne compromet pas les mots de passe). La prochaine étape serait d'activer le chiffrement InnoDB ou d'utiliser un volume chiffré au niveau OS.

Maintenance
Quelle est la différence entre maintenance préventive, évolutive et corrective ?

Préventive : actions planifiées régulièrement pour éviter des problèmes futurs. Exemple : mettre à jour les dépendances chaque mois pour corriger des CVE avant qu'elles soient exploitées. Évolutive : faire évoluer le produit selon de nouveaux besoins. Exemple : ajouter une fonctionnalité de messagerie interne. C'est facturable (estimé en jours, TJM 350€). Corrective : réparer un bug ou un dysfonctionnement existant. Elle a des SLA : un problème critique doit être pris en charge en 2h et résolu en 24h.

Pourquoi GitHub Issues plutôt qu'un outil dédié comme Jira ?

Jira est un outil puissant mais complexe, avec un coût, une configuration importante, et qui vit en dehors du dépôt de code. GitHub Issues est directement intégré au dépôt : un ticket peut référencer un commit (fixes #42 dans un message de commit ferme automatiquement l'issue), une PR peut être liée à une issue, et l'historique complet est dans le même outil que le code. Pour un projet open source ou une petite équipe, c'est plus simple, gratuit, et suffisamment puissant avec les labels et milestones.

Que couvre la garantie de 30 jours sur les évolutions ?

Toute fonctionnalité livrée dans le cadre de la maintenance évolutive est garantie 30 jours contre les régressions directement liées à cette livraison. Concrètement : si on livre une nouvelle fonctionnalité de filtre et que ça casse le système de recherche existant, la correction est incluse sans facturation supplémentaire. En revanche, cette garantie ne couvre pas les bugs préexistants, les problèmes dus à l'infrastructure tierce, ou les modifications de comportement demandées après livraison.

Pourquoi la rotation des clés JWT est-elle trimestrielle ?

La rotation des clés est un principe de sécurité : même si une clé privée a été compromise sans qu'on le sache, la remplacer régulièrement limite la fenêtre d'exposition. Trimestrielle (tous les 3 mois) est un compromis entre sécurité et praticité — trop fréquente, ça génère des opérations manuelles et des risques d'erreur. La rotation consiste à générer une nouvelle paire RSA, mettre à jour les GitHub Secrets, et redéployer. Tous les tokens existants deviennent invalides (TTL max 1h de toute façon), les utilisateurs se reconnectent.

Qu'est-ce que le RPO et le RTO de votre projet ?

RPO (Recovery Point Objective) = 24h : en cas de catastrophe (serveur qui brûle, BDD corrompue), on peut perdre au maximum les données des 24 dernières heures, car la sauvegarde mysqldump est quotidienne. RTO (Recovery Time Objective) = 4h : en cas de panne, l'application doit être de nouveau accessible en moins de 4 heures. On atteint ce RTO grâce aux images Docker versionnées sur GHCR : il suffit de pointer vers le tag précédent et de relancer les conteneurs via SSH — procédure qui prend moins d'une heure.

Architecture & Concepts
Pourquoi avoir 3 dépôts séparés plutôt qu'un seul monorepo ?

Chaque dépôt a son propre cycle de vie, ses propres dépendances, et son propre pipeline CI. Un développeur Flutter n'a pas besoin de télécharger les dépendances PHP pour travailler. La CI de l'app mobile ne bloque pas le déploiement de l'API. Si on met tout dans un monorepo, la configuration devient complexe (filtres de chemin, pipelines conditionnels), et le premier développeur venu qui casse la CI PHP bloque potentiellement tous les autres projets.

Qu'est-ce que le Semantic Versioning et comment vous l'utilisez ?

SemVer est une convention de numérotation MAJEUR.MINEUR.PATCH. MAJEUR = changement incompatible (ex: v1.0.0 → v2.0.0 si on supprime un endpoint API). MINEUR = nouvelle fonctionnalité rétrocompatible (ex: v1.0.0 → v1.1.0 si on ajoute un endpoint). PATCH = correction de bug (ex: v1.0.0 → v1.0.1). Ça permet à n'importe qui — y compris l'app mobile — de savoir si une mise à jour de l'API est sûre à adopter sans changement de code de leur côté.

Pourquoi utiliser Prometheus + Grafana + Loki plutôt qu'une solution SaaS comme Datadog ?

Datadog est excellent mais coûteux (facturation à l'usage, rapidement plusieurs centaines d'euros/mois pour un projet sérieux). Prometheus + Grafana + Loki sont open source, auto-hébergés, et gratuits. La stack PLG (Prometheus/Loki/Grafana) est la référence du marché pour l'auto-hébergement — Grafana peut afficher métriques et logs dans le même dashboard. L'inconvénient : on gère soi-même la configuration et la rétention des données. Pour un projet CESI, c'est aussi l'occasion d'apprendre des outils industriels réels.
